## 1. 背景

SG1210 固件原有**两套不完整的 GUI Form 管理系统**：

| 系统 | 文件 | 问题 |
|------|------|------|
| GUICntr | `GUICntr.cpp/h` | 栈式管理，表单硬编码在 `GFormsList[]`，`MAX_FORMS=8`，扩展性差 |
| FormManager | `FormManager.cpp/h` | 半成品单例，`unordered_map` 动态注册，无栈无 Tick，引用不存在的字段 |

**GForm 统一系统**替换上述两者，提供干净的 C++20 namespace 风格 API。

## 2. 架构设计

```
┌─────────────────────────────────────────────────┐
│  调用层                                          │
│  WinMain.cpp  ──→  gform::KeyEvent()             │
│  GUIMain.cpp  ──→  gform::Init/Tick/PushForm()   │
│  各 Form.cpp  ──→  gform::PushForm/PopForm/SendMsg│
│                ──→  FormRegistrar (静态自注册)     │
├─────────────────────────────────────────────────┤
│  引擎层                                          │
│  gform::                                         │
│  ├─ Init()         导航栈 + 消息队列初始化         │
│  ├─ RegisterForm() 动态注册 (32槽，保留静态注册)    │
│  ├─ PushForm()     压栈 → pInit → pShow          │
│  ├─ ReplaceForm()  弹出旧 → 压入新               │
│  ├─ PopForm()      弹出 → 恢复前一个              │
│  ├─ SendMsg()      向栈顶表单发送消息              │
│  ├─ PostMsg()      延迟消息 (下个 Tick 投递)       │
│  ├─ Tick()         10ms 定时 + 消息队列排空        │
│  └─ KeyEvent()     按键翻译 → SendMsg             │
├─────────────────────────────────────────────────┤
│  平台层                                          │
│  GFormPlatform.h                                 │
│  ├─ platform::Lock   (std::mutex / osMutexId)     │
│  └─ ScopedLock       RAII 自动加锁/解锁           │
├─────────────────────────────────────────────────┤
│  注册层                                          │
│  GFormRegistrar.h  (RAII 静态构造 → RegisterForm) │
└─────────────────────────────────────────────────┘

  ~~ GUICntr.cpp 适配层 ~~  已不再被任何活跃代码调用
```

### 核心设计决策

| 决策 | 理由 |
|------|------|
| **namespace 而非 singleton** | 无构造/析构顺序问题，语义最简 |
| **固定数组而非动态容器** | 无堆分配，适合嵌入式 Cortex-M4 |
| **保留 GWinForm 4函数指针** | 现有表单无需修改结构体声明 |
| **PlatformLock 抽象** | 业务逻辑零 `#ifdef __vmSIMULATOR__` |
| **FormRegistrar RAII** | 静态初始化自动注册，增删 .cpp 自动生效 |
| **栈式导航 (push/replace/pop)** | 适合嵌入式 UI 层级结构 |
| **Init() 不清空注册表** | 静态初始化期注册的表单在 Init() 后依然有效 |

## 4. API 参考

### 4.1 类型定义

```cpp
using FormId = uint16_t;

enum class FormTransition : uint8_t {
    kPush,      // 压栈并显示
    kReplace,   // 弹出当前 → 压入并显示
    kPop        // 弹出当前 → 恢复前一个
};

struct FormRecord {
    const GWinForm* callbacks;   // Init/Show/Close/Msg 四个回调
    const char*     name;        // 调试名称 (UTF-8)
    uint16_t        flags;       // 动画/过渡标志位 (预留)
};

// ID 常量
constexpr uint16_t WID_SplashForm     = 100;
constexpr uint16_t WID_MenuForm       = 101;
constexpr uint16_t WID_MainForm       = 102;
constexpr uint16_t WID_EventListForm  = 103;
// ... 等 (定义在 GForm.h，与旧 GUICntr.h 兼容)
constexpr uint16_t kFormIdInvalid = 0xFFFF;

// 容量限制
constexpr size_t kMaxForms = 32;   // 注册表容量
constexpr size_t kMaxStack = 16;   // 导航栈深度
```

### 4.2 生命周期

```cpp
void gform::Init();          // 初始化导航栈和消息队列 (不清空注册表)
void gform::Tick();          // 10ms Tick — 排空延迟消息 + 投递 GM_TIMER_TICK
size_t gform::Run();         // 返回当前栈深度
```

### 4.3 注册

```cpp
void gform::RegisterForm(FormId id, const GWinForm* form, const char* name);
const FormRecord* gform::FindForm(FormId id);
void gform::UnregisterForm(FormId id);

// RAII 自动注册 — 在 Form.cpp 文件作用域使用
#include "GFormRegistrar.h"
static const gform::FormRegistrar kReg(WID_SPLASH, &FSplashForm, "Splash");
// FormRegistrar 构造函数在 main() 前调用 RegisterForm()
// gform::Init() 不清空注册表，故注册在 Init() 后依然有效
```

### 4.4 导航

```cpp
void gform::PushForm(FormId id, const void* para = nullptr);
void gform::ReplaceForm(FormId id, const void* para = nullptr);
void gform::PopForm();                          // 忽略参数，弹出后恢复前一个
void gform::CloseCurrentForm();
FormId gform::GetCurrentFormId();
```

### 4.5 消息

```cpp
void gform::SendMsg(uint16_t msgId, uint16_t param = 0, int32_t value = 0);
void gform::SendMsgPtr(uint16_t msgId, uint16_t param, const void* data);
void gform::PostMsg(uint16_t msgId, uint16_t param = 0, int32_t value = 0);
void gform::BroadcastMsg(uint16_t msgId, uint16_t param = 0, int32_t value = 0);
void gform::KeyEvent(uint32_t key, uint32_t pressedCnt);
```

```cpp

```

## 8. CSG 图像编解码集成

### 8.1 背景

CSG (Compact Scalable Graphic) 是银网科技自研的图像压缩格式（v1.5），编解码器位于 `Middlewares/emWin/csgimage/`。解码器在模拟器构建中已编译，但此前未注册到 emWin 的图像绘制管线。

### 8.2 集成方式

通过 `GUI_DrawPicture()` 调度函数接入，无需修改 emWin 内核：

```cpp
// GUIPicture.cpp — GUI_DrawPicture()
void GUI_DrawPicture(const TGUIPicture *pPic, int x0, int y0) {
    switch (pPic->Type) {
        case ID_BITMAP: LCDX_Bitmap_Draw(...);  break;   // 已有
        case ID_CSG:    /* CSG decode + draw */  break;   // 新增
        case ID_JPG:    GUI_JPEG_Draw(...);      break;   // 已有
    }
}
```

### 8.3 数据流

```
TGUIPicture (Type=ID_CSG, pData=CSG bytes, Size=N)
  │
  ▼  GUI_DrawPicture(&pic, x0, y0)
  │
  ▼  case ID_CSG:
CSGDecoder::DecodePicture(data, len)
  │  ① 解析 CSGPicture header (16B): magic "Vx", width, height, colorMode, CRN, CAS
  │  ② CRC16 校验
  │  ③ 根据 CAS 选择解压器: RLE / MiniLZ77 / Huffman / DEFLATE(MiniLZ77+Huffman)
  │  ④ 调色板索引 → CRM 原始色深数据
  │  ⑤ CRM → RGBA 转换 + 透明色处理 (CRI=0 或 palette[0] 匹配)
  ▼
DecoderResult { .pixels=RGBA[W*H*4], .width, .height, .colorMode, ... }
  │
  ▼  逐像素遍历:
  │    if (alpha > 0) { GUI_SetColor(r,g,b); GUI_DrawPixel(x0+x, y0+y); }
  ▼
LCD 显示
```

### 8.4 修改文件

| 文件 | 变更 |
|------|------|
| `Middlewares/emWin/user/GUIPicture.cpp` | 新增 `#include "CSGDecoder.h"` + `case ID_CSG:` 分支 |
| `Application/GUI/GPSplashForm.cpp` | 新增 `#include "Graphics/x128.h"` + `GUI_DrawPicture(&picx128csg, ...)` 测试绘制 |

### 8.5 支持的 CSG 格式

| 参数 | 支持范围 |
|------|----------|
| 色深 (ColorMode) | RGB565, BGR565, RGB666, RGB888, RGB8888 |
| 调色板 (CRN) | 0 (真彩), 2, 4, 8, 16, 32, 64, 128 |
| 压缩 (CAS) | None, RLE, MiniLZ77, Huffman, DEFLATE |
| 压缩等级 (CAL) | 0–7 |
| 分辨率 | 10×10 到 256×256 |
| 图集 (Atlas) | 支持多图 CSG 文件 |

## 9. 踩坑记录

### 9.1 `struct` vs `typedef struct` 前向声明冲突

**问题**: `GWinForm` 定义为 `typedef struct tagGWinForm { ... } GWinForm;`，使用 `struct GWinForm;` 前向声明会导致 MSVC 报 `C2371: 重定义；不同的基类型`。

**修复**: 在 `GForm.h` 和 `GFormRegistrar.h` 中直接 `#include "GWinTypes.h"`，不使用前向声明。

### 9.2 Windows API 宏冲突

**问题**: `<Windows.h>` 将 `SendMessage` 宏展开为 `SendMessageA`，导致 `gform::SendMessage()` → `gform::SendMessageA()`，编译失败。`PostMessage` 同理。

**修复**: 重命名为 `gform::SendMsg()` / `gform::PostMsg()`。

### 9.3 静态初始化顺序

**问题**: `FormRegistrar` 在 `main()` 前构造调用 `RegisterForm()`，但 `gform::Init()` (在 `main()` 后调用) 原本会清零注册表。

**修复**: `Init()` 不再清零注册表，仅重置导航栈和消息队列。

### 9.4 空数组编译错误

**问题**: `s_formTable[] = {}` 在 MSVC 中产生零大小数组，报 `C2466: 不能分配常量大小为 0 的数组`。

**修复**: 添加哨兵 `{ 0, nullptr, nullptr }` + `RegisterAllForms()` 中 null 检查。

### 9.5 GB2312/UTF-8 编码与 C4819 警告

**问题**: 旧文件使用 GB2312 编码，MSVC 代码页 936 报 C4819 警告。`sed` 可能破坏编码导致编译错误。

**修复**: 新文件统一使用 UTF-8 with BOM。旧文件编码转换留待 Phase 4。

### 9.6 VS 项目 post-build 路径错误

**问题**: Post-build 步骤 `copy ShivaVG.dll exe\` 目标目录 `exe\` 不存在。

**修复**: 改为 `copy ShivaVG.dll Build\`。

### 9.7 静态初始化崩溃 (Segfault)

**问题**: 模拟器启动时在 `main()` 之前 SIGSEGV 崩溃。`FormRegistrar` 静态构造器调用 `RegisterForm()` → `ScopedLock(s_lock)` → `m_mutex.lock()`。`s_lock` 是全局 `gform::platform::Lock`（内含 `std::mutex`），其构造顺序与其他翻译单元的静态初始化存在竞争。

**修复**:
1. 模拟器 `platform::Lock::Acquire/Release` 改为空操作（模拟器单线程，无需真锁）
2. 表单注册回退到 `s_formTable` 集中注册（`GUICntr::RegisterAllForms`），由 `GUIStart()` 在 `main()` 之后调用
3. `FormRegistrar` 静态注册对象暂禁用

### 9.8 MSBuild PDB 文件锁定

**问题**: 多核并行编译时，多个 `cl.exe` 进程争用同一个 `.pdb` 文件，报 `C1041`。

**临时修复**: 使用 `-maxcpucount:1` 单线程编译。长期方案：添加 `/FS` 编译选项。

### 9.9 Git 中 Sim 目录为子模块

**问题**: `git add` 时 Sim 目录未被跟踪，`git checkout` 无法恢复 Sim 下的文件。

**原因**: Sim 目录内有独立的 `.git` 目录（可能是 VS 或先前操作残留）。

**修复**: `rm -rf Sim/.git` 将其纳入主仓库。

## 17. MiniLZ77 环形窗口索引 (2026.06.26)

### 背景

MiniLZ77 解码器使用 `uint8_t window[CSG_DEC_WINDOW_BYTES]` 作为滑动窗口历史缓冲。窗口以绝对像素位置索引：`win[outPos] = cri`、`cri = win[ref + j]`。当图片总像素超过 `CSG_DEC_WINDOW_BYTES`（8192）时，绝对索引越界。

虽然 MiniLZ77 最大 lookback=128 像素，但 `outPos` 从 0 累加到 totalPixels（如 256×256=65536），超过 8192 后数组越界。

### 修复

所有窗口访问改为环形（模除）索引：

```cpp
// 修复前
win[outPos] = cri;
cri = win[ref + j];

// 修复后
win[outPos % CSG_DEC_WINDOW_BYTES] = cri;
cri = win[(ref + j) % CSG_DEC_WINDOW_BYTES];
```

涉及 9 处访问点：
- `DecodeMiniLZ77` 辅助函数（DEFLATE→MiniLZ77 路径）：5 处
- `CsgDecodePixels` 内联 MiniLZ77（CAS=3 直接路径）：4 处

### 安全性分析

- MiniLZ77 最大 lookback=128，最大 match=258，工作集跨度 ≤ 128+258 = 386 像素
- CSG_DEC_WINDOW_BYTES=8192 >> 386，新写入不会覆盖仍被引用的历史数据
- `pendRef` 存储绝对像素位置，恢复时模除索引正确回绕
- `CSG_DEC_WINDOW_BYTES=8192` 是 2 的幂，编译器将 `% 8192` 优化为 `& 8191`

### 测试结果

- 256×256 大图 2000+ 次循环无花屏 ✅
- 全部 11 张 ImageRes 图正常 ✅

## 27. CSG DrawPicture — pClipRect 参数替代 emWin ClipRect (2026.06.30)

### 问题

`CSG_DrawPicture` 在 MCU 绕过 emWin 直写 LCD(FSMC)，无法依赖 emWin 的 `GUI_SetClipRect`/`GUI_GetClipRect`：
1. `GUI_GetClipRect` 取不到准确的裁剪区域
2. emWin 控件（`GUI_FillRect`、`GUI_DrawRoundedRect`、`GUI_DispStringInRect`）会修改 ClipRect，泄漏到后续 CSG 调用
3. 尝试在 Form 入口 `SetFullClip()` 无效——emWin 操作在中间会覆盖它

### 方案演进

| 尝试 | 方法 | 结果 |
|------|------|------|
| 1 | MemDev 快照还原 | Win32 Sim 驱动不支持往返 |
| 2 | `GUI_SetClipRect` + CSG 读取 `GUI_GetClipRect` | `GUI_GetClipRect` 返回值不准确 |
| 3 | Form 入口 `SetFullClip()`（GUIMisc） | emWin 控件中途覆写 clip |
| **4** | **`pClipRect` 参数直接传入** | ✅ **最终方案** |

### 最终方案

`CSG_DrawPicture` 新增 `const GUI_RECT* pClipRect = nullptr` 参数：

```cpp
void CSG_DrawPicture(const TGUIPicture* pPic, int x0, int y0,
                     int picIndex = 0, int saturation = 100,
                     const GUI_RECT* pClipRect = nullptr);
```

| `pClipRect` | 行为 |
|-------------|------|
| `nullptr`（默认） | 全屏绘制，零裁剪开销 |
| `&rect` | 仅绘制 rect 内像素（行列裁剪） |

MCU 路径：
- `nullptr` → 快路径，无裁剪判断
- `&rect` → 行级跳过（Y 范围）+ 列级裁剪（iClipStart/iClipEnd）+ 逐像素跳过

### 调用方职责

| 场景 | 文件 | 调用 |
|------|------|------|
| 正常全屏绘制 | 所有 Form | `CSG_DrawPicture(pPic, x, y)` — 省略 clip 参数 |
| 局部擦除选取框 | GPMenuForm `_EraseSelArea` | `CSG_DrawPicture(bkg, 0, 0, 0, 100, &rClip)` |

### MCU ClipRect 教训

1. **不能绕过 emWin 又依赖 emWin** — CSG 直写 LCD 就必须有自己的裁剪机制
2. **emWin 控件会修改 ClipRect** — `GUI_FillRect`/`GUI_DrawRoundedRect`/`GUI_DispStringInRect` 调用后 clip 不可靠
3. **默认参数保持简洁** — `pClipRect = nullptr` 使 95% 的调用点无需改动
4. **局部擦除是唯一需要裁剪的场景** — `_EraseSelArea` 是唯一传 `&rClip` 的调用点

## 33. DFT80 3-5次谐波扩展 (2026.07.01)

### 算法

80 点多项式 DFT 原理：
- 预存 20 个余弦值 `DFT80P0..P19 = cos(2π×n/80)`
- 对谐波 k，样本 n 的余弦表索引 = `(k×n) % 80`，利用 cos 四象限对称性将 80 个样本合并为 ≤20 个乘法项
- 虚部 `-sin` 通过余弦表偏移 N/4=20 个样本计算

### H3/H4/H5 特性

| 谐波 | gcd(80,k) | 周期 | 实部项数 | 每项样本数 |
|------|-----------|------|---------|-----------|
| H3 | 1 | 80 | 20 | 4 |
| H4 | 4 | 20 | 5 | 16 |
| H5 | 5 | 16 | 4 | 20 |

### 实现的 3 套机制

| 机制 | 消息 | 引擎行为 |
|------|------|----------|
| Form 激活/失活通知 | `GM_FORM_ACTIVATED`(18) / `GM_FORM_DEACTIVATED`(19) | Push/Replace/Pop 时通知相关 Form |
| 关闭前查询 | `GM_CLOSE_QUERY`(130) | Pop 前查询当前 Form，`Data.v>0` 则 veto 中止弹出 |
| 空闲超时查询 | `GM_IDLE_EXPIRE`(22) | GUITick 超时前 `gfc::QueryIdleExpire()` 询问，Form 可 veto |

### 导航回调时序（实际）

| 操作 | 时序 |
|------|------|
| **Push** | DEACTIVATED(旧栈顶) → pInit(新) → pShow(新) → ACTIVATED(新) |
| **Replace** | DEACTIVATED(旧) → 弹栈 → pClose(旧) → pInit(新) → pShow(新) → ACTIVATED(新) |
| **Pop** | CLOSE_QUERY(当前,可veto) → DEACTIVATED(当前) → 弹栈 → pClose(旧) → pShow(下层) → ACTIVATED(下层) |

新增匿名 namespace 辅助函数 `NotifyForm()` / `QueryForm()`（锁内调用，空指针守卫）。
`ActivateForm` 末尾统一发 ACTIVATED，供 Push/Replace 复用。

### 设计决策

- **CLOSE_QUERY 仅保护 PopForm**：Replace 为系统强制（空闲超时回 Main），不查询。veto 协议 `Data.v>0` 与 IDLE_EXPIRE 一致。
- **IDLE_EXPIRE 查询链**：`GUICntr::GUITick` 超时分支原直接 `ReplaceForm(MainForm)`，改为先 `gfc::QueryIdleExpire()`；Form veto（如编辑中 isModified）则保留。`uKeyIdle=0` 仍先置零避免重复触发。
- **CloseCurrentForm 顺手修复**：补 `s_stackTop==0` 空栈保护（原潜伏 `s_stack[-1]` 越界），并补对称的 DEACTIVATED/ACTIVATED 通知（不加 CLOSE_QUERY，强制关闭语义）。

### 行为兼容性

现有活跃 Form（Splash/Menu/Main/EventBrowser 等）的 `_OnMessage` switch 命中 default→break，忽略新消息。三套机制表现为"就绪、无行为变化"，符合 P0 交付定义。Splash 超时在 `GM_TIMER_TICK` 处理中调 `ReplaceForm` → 同步重入发 DEACTIVATED，MCU 锁 `osMutexRecursive`、Sim 锁 no-op，重入安全。

## 51. GDataListForm 滚动重绘 Bug 修复 (2026.07.09)

### 现象

按键 UP/DOWN 移动游标至视窗顶端/底端（触发滚动）时，列表内容错位、重绘不正确。

### 根因

`_OnKeyDown` 的滚动判定条件错误：
```cpp
if (old < m_State.uTopItem || old >= m_State.uTopItem + DL_VISIBLE) {
    _UpdateList();        // 仅当旧游标在新视窗外才全刷
} else {
    _DrawRow(old); _DrawRow(m_State.uCurItem);  // 否则只刷两行
}
```
该条件检查"旧游标是否在新视窗外"。但触底/触顶滚动时，`_EnsureCursorVisible` 改变 `uTopItem` 后，**旧游标常仍在视窗内**（如 DOWN 触底：top 0→1，旧游标 10 仍在 [1,12) 内），条件为假 → 只重绘两行。而实际上整屏已滚动一行（每个索引的像素位置都变了），未重绘的行留下旧内容 → 错位。

### 修复

改为判定 `uTopItem` 是否变化：变化则全刷，否则只刷新旧行。
```cpp
uint16_t oldTop = m_State.uTopItem;
// ... move cursor, _EnsureCursorVisible ...
if (m_State.uTopItem != oldTop) {
    _UpdateList();
} else {
    _DrawRow(old);
    _DrawRow(m_State.uCurItem);
}
```
UP/DOWN 两处同改。

## 55. GDataListForm 导航崩溃 + 体验改进 (2026.07.09)

### 1. 崩溃根因与修复

## 59. GDataListForm 滚屏顶行修复 + 长按连续滚动 (2026.07.09)

### 1. 滚屏后按 UP 首行（组头）不显示

**根因**：游标跳过组头行（`_NextReg` 只落 kReg），故游标无法到达 row 0（group1 头）。`_EnsureCursorVisible` 仅在游标移出视窗时滚动，导致 `uTopItem` 一旦 >0（DOWN 滚屏时设为 `uCurItem-DL_VISIBLE+1`）就回不到 0——group1 头被永久遮挡，顶行恒为第二行（首寄存器）。

**修复**：`_OnKeyDown` UP/DOWN 解耦"游标移动"与"视口滚动"——
- 游标可移至相邻寄存器：移动游标 + `_EnsureCursorVisible`。
- 游标已在首/末寄存器（无法移动）但视口可滚：视口单行滚动（`--uTopItem`/`++uTopItem`），露出其上/下的行（如组头）。
- 都不可：break。

这样按 UP 到首寄存器后继续按 UP，`uTopItem` 从 1→0，group1 头重新显示于顶行。

### 2. 长按上下键连续响应

Sim 的 `WinMain` WM_KEYDOWN 重复计数（`bPrevDown`）→ `gfc::KeyEvent(wParam, nKeyPressCnt>1)` → `GM_KEYPRESS`。DataListForm 原仅处理 `GM_KEYDOWN`/`GM_KEYUP`，未处理 `GM_KEYPRESS`，故长按不重复。

**修复**：`_OnMessage` 新增 `case GM_KEYPRESS: _OnKeyDown(pMsg->Param)`，与 `GM_KEYDOWN` 同路径。首按 `GM_KEYDOWN` 触发一次，持续按住 `GM_KEYPRESS` 重复 UP/DOWN（及 LEFT/RIGHT，受 PushForm 重入保护/PopForm 栈底保护，重复无害）。ESC 仅在 `GM_KEYUP` 响应，不重复。

|------|------|------|
| `GPDataListForm.cpp` | UP/DOWN 解耦视口边缘滚动（修首行组头遮挡）；`GM_KEYPRESS` 路由至 `_OnKeyDown`（长按连续） | V1.05→V1.06 |

## 60. GDataListForm 按键节流（MCU 单步行）(2026.07.09)

### 问题

MCU 路径按键响应太快，单次点按移多行，无法单步上下移一行。

### 根因

MCU 键盘驱动 `KeyBoard_Read` 每 10ms 扫描一次，按下后持续产生 `KEYST_PRESS`→`GM_KEYPRESS`（每 10ms 一次，**无初始延迟**）。§59 加的 `GM_KEYPRESS` 处理全部响应 → 一次点按（~100ms）≈ 1 DOWN + ~9 PRESS = 10 行。Sim 因 Windows 自带 ~500ms 初始延迟不明显。

### 修复

`_OnKeyDown` 改为 `_OnKey(key, bRepeat)`，加节流（与驱动扫描时序解耦）：
- `GM_KEYDOWN`（首按）：立即响应，`uLastKeyTick=now`，`uKeyRepeat=0`。
- `GM_KEYPRESS`（持续）：仅当距上次响应 ≥ 阈值才响应，并更新计时；否则丢弃。
  - 阈值：首次重复 `DL_KEY_REPEAT_INIT_MS=300`，其后 `DL_KEY_REPEAT_MS=120`。
- `GM_KEYUP`：`uKeyRepeat=0`（下次点按为新的首按）。

效果：点按（<300ms）= 恰好 1 行（DOWN 后的 PRESS 全被初始延迟挡掉）；长按 = 0ms 首行、300ms 后第 2 行、此后每 120ms 一行。Sim 同样适用（Windows 重复经节流归一为 120ms）。

| 文件 | 变更 | 版本 |
|------|------|------|
| `GPDataListForm.cpp` | `_OnKey` 节流（初始 300ms + 重复 120ms）；`GM_KEYDOWN`/`KEYPRESS` 分路；`_OnKeyUp` 重置重复计数 | V1.06→V1.07 |

## 64. DevEvtMgr 审查修复：空指针守卫 + FieldData 越界 + 并发互斥（方案 A）(2026.07.11)

### 背景

审查 `DevEvtMgr` 发现 3 类问题：#1 空指针解引用（非 assert 构建）、#2 FieldData 越界写、#3 多任务并发无锁。`AppendEvent` 多用于硬件中断（CoilCtrl，目前注释）。按方案 A 修复：ISR 经队列延迟到任务级，事件缓冲区用互斥保护。

### 修复

**#1 空指针守卫**：`AppendEvent`/`AppendOpLog` 在 `DevIntf_GetRegInfo`/`GetEvtProp` 后加显式 `if (nullptr == p) return;`（assert 仅调试，非唯一防线）。

**#2 FieldData 越界**：`AppendEvent` 现场数据循环上限 clamp 为 `uIdx < NbrOfData && uIdx < NUM_EVENT_DATA`（防止 `NbrOfData > NUM_EVENT_DATA` 越界写 `FieldData`/读 `FieldDataReg`）。

**#3 并发互斥（方案 A）**：
- 事件缓冲区（`EventLogCache`/`DispEventLogCache`）加 MCU 互斥 `s_lock`（`osMutexNew`，递归+优先级继承），`EVTMGR_LOCK/UNLOCK` 宏（Sim no-op）。
- `AppendEvent`/`AppendOpLog`/`SaveEvent`/`GetNewEventCount`/`Fetch`/`Search`/`ClearNewEventMarks` 均在锁内访问缓冲区。
- `SaveEvent` 锁内遍历+`FIX_SaveEvtLogItem`（FeRAM 写快，可持锁）；`Fetch`/`Search` 锁内找项+清标志后解锁返回（返回指针指向环形缓冲内部，注明调用方应立即拷贝）。
- ISR 延迟路径：`EVTMGR_PostEvent`（任务级 `xQueueSend`）/ `EVTMGR_PostEventFromISR`（ISR `xQueueSendFromISR`）入队 `EvtItem{uRegNum,uAction}`；`EVTMGR_DrainEvents`（appTask 循环）排空 → `AppendEvent`（任务级）。FreeRTOS `QueueHandle_t`（32 槽），`EVTMGR_Init` 创建。
- Sim：`PostEvent` 直连 `AppendEvent`（单线程），`DrainEvents` no-op，锁 no-op。
- appTask（`TaskApp.cpp`）循环每周期调 `EVTMGR_DrainEvents()`。
- 当前活跃调用点（COM/DevFixed/DevIntf/TaskApp）均为任务级，直接 `AppendEvent`（互斥安全）；未来 CoilCtrl ISR 重启用 `PostEventFromISR`。

## 67. GLogListForm 闪烁问题修复与验证 (2026.07.11)

### 问题描述

MCU 液晶屏上 GLogListForm 行表出现周期性闪烁，用户反馈影响使用体验。

**根因分析**：
- `_OnTick()` 函数每 1000ms（`LL_REFRESH_MS`）调用 `_RefreshValues()` 进行定时刷新
- 即使没有用户交互，表单内容也在周期性重绘，导致 LCD 闪烁
- 违反了"GLogListForm 只需要响应交互操作，更新显示即可"的设计原则
### 修复方案

#### 代码变更（GLogListForm.cpp）

**移除定时刷新机制**：
- ❌ 删除 `LL_REFRESH_MS` 常量（原为 1000ms）
- ❌ 删除 `uLastRefresh` 字段（定时器时间戳）
- ❌ 清空 `_OnTick()` 函数（取消自动刷新）
- ❌ 移除 `_Show()` 中的定时器初始化

**保留交互逻辑**：
- ✅ `uLastKeyTick` 字段（UP/DOWN 按键重复节流）
- ✅ `_RefreshValues()` 函数（分类切换时手动调用）
- ✅ 所有用户交互（按键、触屏、分类切换）

#### 新行为

| 操作 | 触发条件 | 行为 |
|------|----------|------|
| 进入表单 | _Show 回调 | 一次性加载日志数据 + 完整绘制 |
| 上/下按键 | 用户按下 | 更新光标位置 → 调用 _UpdateList() → 绘制 |
| 左/右滑动 | 用户滑动 + 超过阈值 | 分类切换 → _ChangeType() → 重新加载 → 绘制 |

## 68. CSG 解码器 bitCount 初始化修复 — indexed color 4列错位 (2026.07.13)

### 问题描述

CSG 流式解码器在 `/cas None` 模式下，使用 indexed color（CRN=4/8）时，出现 **4列错位** 现象：
- **表现**：每行的最后 4 个像素出现在下一行的前 4 列
- **影响范围**：所有使用 `/cas none` + indexed color 的图片
- **测试用例**：
  - `LV_CrossMark16x16Green.png /crn 8 /cas none`
  - `MU_Home24x22.png /crn 4 /cas none`
  - `MU_Item32x32_01.png /crn 4 /cas none`

### 根本原因

**bitCount 字段的双重语义**：`CSGDecoderState::bitCount` 在不同压缩算法中有不同含义：

| 压缩算法 | bitCount 含义 | 初始值应为 |
|---------|--------------|-----------|
| CAS=0 (None) | bit accumulator，bit-unpacking 时的已填充位数 | **0** (空 buffer) |
| CAS=1 (RLE) | bit accumulator | **0** (空 buffer) |
| CAS=2 (DEFLATE) | 由 Huffman 管理，此处不使用 | **0** |
| CAS=3 (MiniLZ77) | token index (0-7) 或 8=需要读取新控制字节 | **8** (读新 ctrl) |

**错误的初始化代码**（CSGDecoder.cpp:341 修复前）：
```cpp
state->bitCount = 8;  // ❌ 对所有压缩算法都设为 8
```

对于 CAS=0 的 indexed color 模式：
1. `DecodeNone` 在 bit-unpacking 时，从 `bitCount=8` 开始
2. 第一次进入循环，认为 buffer 中已有 8 bit 数据
3. 实际 `bitBuf=0`，这 8 bit 全是无效数据
4. 导致前 8 个 index 被错误解析，产生 4 列错位（8 index ÷ 2 bpi = 4 列）

### 修复方案

#### 1. 条件初始化 bitCount

**CSGDecoder.cpp:364**（核心修复）：
```cpp
// 修复前
state->bitCount = 8;

// 修复后
state->bitCount = (static_cast<int>(CompressAlgorithm::kMiniLZ77) == state->cas) ? 8 : 0;
```

**逻辑**：
- 如果是 MiniLZ77（CAS=3）→ `bitCount=8`（需要读取第一个 ctrl 字节）
- 其他算法（CAS=0/1/2）→ `bitCount=0`（bit buffer 为空）

#### 2. 替换 magic number 为枚举常量

提升代码可维护性，避免 magic number 3：

**CSGDecoder.cpp:873**（MiniLZ77 dispatch 判断）：
```cpp
// 修复前
if (3 == state->cas) {

// 修复后
if (static_cast<int>(CompressAlgorithm::kMiniLZ77) == state->cas) {
```

**CSGDecoder.cpp:980-987**（CAS switch 分支）：
```cpp
// 修复前
case 0:  // None
case 1:  // RLE
case 2:  // DEFLATE

// 修复后
case static_cast<int>(CompressAlgorithm::kNone):
case static_cast<int>(CompressAlgorithm::kRLE):
case static_cast<int>(CompressAlgorithm::kDEFLATE):
```

### 技术细节

#### DecodeNone bit-unpacking 逻辑

**CSGDecoder.cpp:489-509**：
```cpp
while (dec < toDecode) {
    // Refill bitBuf when empty
    while (bitCount < bpi && ip < ipEnd) {
        bitBuf = (bitBuf << 8) | input[ip++];
        bitCount += 8;
    }
    
    if (bitCount >= bpi) {
        // Extract CRI index
        uint8_t cri = (bitBuf >> (bitCount - bpi)) & bpiMask;
        bitCount -= bpi;
        // Write to output with palette lookup
        WriteCrmPixel(output, dec, bpc, pal, cri);
        ++dec;
    } else {
        break;  // Not enough bits, need more data
    }
}
```

**关键点**：
- `bitCount` 表示 `bitBuf` 中已填充的有效位数
- 初始必须为 0，否则会误认为 buffer 中有数据
- 每次从 input 读取 8 bit 时，`bitCount += 8`
- 每次提取 bpi 个 bit 后，`bitCount -= bpi`

#### MiniLZ77 tokenIdx 语义

**CSGDecoder.cpp:878-913**：
```cpp
int tokenIdx = state->bitCount;  // 0-7: current token, 8: need new ctrl
while (dec < toDecode) {
    if (8 <= tokenIdx) {
        ctrl = input[ip++];  // Read new control byte
        tokenIdx = 0;
    }
    
    if (0 != (ctrl & (1u << tokenIdx))) {
        // Match token: read distance & length
    } else {
        // Literal token: read raw byte
    }
    
    ++tokenIdx;
}
```

**关键点**：
- `tokenIdx` = 当前 token 在 ctrl 字节中的位索引（0-7）
- `tokenIdx` = 8 是 sentinel 值，表示需要读取新的控制字节
- 初始必须为 8，确保第一次循环就读取 ctrl
